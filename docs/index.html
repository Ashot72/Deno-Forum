It is very likely that Deno will become the new next-generation Node.js. While both Deno and Node.js have the same purpose, their working mechanisms differ.

Deno https://deno.land/ is a simple, modern and secure runtime for JavaScript and TypeScript that uses V8 and is built in Rust.

·        Secure by default. No file, network, or environment access, unless explicitly enabled.

·        Supports TypeScript out of the box.

·        Ships only a single executable file.

·        Has built-in utilities like a dependency inspector (deno info) and a code formatter (deno fmt)

We use Deno.writeFile to write data into the file. After running deno run file.ts command we get PersmissionDenied error which suggests to run again with the –allow-write flag.

This is actually a key feature of Deno. Deno is secure by default. This means that when we run a script with Deno, that script does not by default have all possible permissions.

For example, something like reading or writing files, or sending network requests. These are operations that are always unlocked and possible with Node. Any Node script can do anything.

When you execute a JavaScript file with Node, that file is able to do everything. It could delete all files on your system and nothing will stop it from doing that. Therefore,

when you execute code with node, you have trust that code because code could potentially do a lot of bad things. Deno by default, is not allowed to do everything. By default, when you execute

code with Deno, that code may not write to files, read files, send or get HTTP requests. That is the reason when we tried to write a file it is denied by default, as part of Deno's security model.

When we want to write a file, we have to set the appropriate permissions.

 

Figure 9

In Deno there is no concept of a package manager as external modules are imported directly into local modules. This raises the question

of how to manage remote dependencies without a package manager. In big projects with many dependencies, it will become very hard

and time consuming to update modules if they are all imported individually into individual modules. The standard practice for solving

this problem in Deno is to create deps.ts by convention. With all dependencies centralized in deps.ts, managing this becomes easier.

Let's say your module depends on remote module https://some.url/a.ts. When you compile your module for the first-time a.ts is retrieved.

It will remain this way until you run your module on a new machine (day in production) or reloaded the cache

(through deno cache –reload for example). But what happened if the content in the remote url https://some.url/a.ts is changed? This could

lead to your production module running with different dependency code that your local module. Deno's solution to avoid this is to use

integrity checking and lock files.

 

Figure 28

--lock=lock.json tells Deno what the lock file to use is, while the –lock-write is used to output dependency hashes to the lock file. A lock.json is storing

a hash of the file against the dependency.

 

Figure 29

Collaborator on another machine will download the project's dependencies into the machine's cache by checking the integrity of each resource.

When Deno was initially launches, the creator stated that one of its objectives was to be able to ship Deno code as a single binary,

something similar to what golang does. What happens when you compile Deno code to a binary is that all the runtime and code

is included in that binary, making it self -sustanable. Once you have compiled it, you can just send this binary anywhere, and you will

be able to execute it.
